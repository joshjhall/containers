# Container Image Signing and Attestation

This project signs all release container images using
[Cosign](https://github.com/sigstore/cosign) keyless signing and attaches
SLSA provenance and SBOM attestations for supply chain security.

## Overview

Every tagged release (`v*`) triggers the following pipeline:

1. **Build** — Multi-variant container images are built and pushed to GHCR
1. **Sign** — Each image is signed with Cosign (keyless OIDC via Fulcio)
1. **Attest provenance** — SLSA v0.2 provenance is generated and attached
1. **Attest SBOM** — CycloneDX SBOM is generated and attached as attestation
1. **Record** — All signatures and attestations are recorded in the Rekor
   transparency log

## How Keyless Signing Works

Cosign keyless signing eliminates the need to manage signing keys:

1. GitHub Actions requests an OIDC token from GitHub's token service
1. The token is exchanged with **Fulcio** (Sigstore CA) for a short-lived
   signing certificate bound to the workflow identity
1. The image is signed with the ephemeral key
1. The signature and certificate are recorded in **Rekor** (transparency log)
1. Verifiers check the signature against the Rekor entry and Fulcio root

**Identity binding**: The certificate contains the GitHub Actions workflow
identity (`https://github.com/OWNER/REPO/.github/workflows/ci.yml@refs/tags/vX.Y.Z`),
ensuring signatures can only come from the project's CI pipeline.

## SLSA Compliance

[SLSA](https://slsa.dev/) (Supply-chain Levels for Software Artifacts) is a
framework for ensuring the integrity of software artifacts.

### Current Level: SLSA Level 2

| Requirement                    | Status | Implementation                                   |
| ------------------------------ | ------ | ------------------------------------------------ |
| Source — version controlled    | Yes    | Git repository                                   |
| Build — scripted build         | Yes    | GitHub Actions workflow                          |
| Build — build service          | Yes    | GitHub-hosted runners                            |
| Build — build as code          | Yes    | Workflow defined in `.github/workflows/ci.yml`   |
| Provenance — available         | Yes    | SLSA provenance attestation attached to images   |
| Provenance — authenticated     | Yes    | Signed with Cosign keyless (Fulcio + Rekor)      |
| Provenance — service generated | Yes    | Generated by GitHub Actions, not user-controlled |

### Provenance Contents

The SLSA provenance attestation includes:

- **Builder identity**: GitHub Actions runner
- **Source repository**: GitHub repository URL and commit SHA
- **Build configuration**: Workflow file and job reference
- **Build timestamp**: When the image was built
- **Materials**: Input artifacts (source code reference)

## Verifying Images

### Quick Verification

```bash
# Verify signature, provenance, and SBOM
./bin/verify-image-signature.sh ghcr.io/OWNER/REPO:v1.0.0-minimal

# View full provenance chain
./bin/track-provenance.sh ghcr.io/OWNER/REPO:v1.0.0-minimal
```

### Manual Cosign Verification

```bash
# Install cosign
# https://docs.sigstore.dev/cosign/system_config/installation/

# Verify signature
cosign verify \
  --certificate-identity-regexp='^https://github.com/OWNER/REPO' \
  --certificate-oidc-issuer=https://token.actions.githubusercontent.com \
  ghcr.io/OWNER/REPO:v1.0.0-minimal

# Verify SLSA provenance
cosign verify-attestation \
  --certificate-identity-regexp='^https://github.com/OWNER/REPO' \
  --certificate-oidc-issuer=https://token.actions.githubusercontent.com \
  --type=slsaprovenance \
  ghcr.io/OWNER/REPO:v1.0.0-minimal

# Verify SBOM attestation
cosign verify-attestation \
  --certificate-identity-regexp='^https://github.com/OWNER/REPO' \
  --certificate-oidc-issuer=https://token.actions.githubusercontent.com \
  --type=cyclonedx \
  ghcr.io/OWNER/REPO:v1.0.0-minimal
```

## Generating SBOMs

SBOMs are generated in two ways:

1. **CI pipeline** — Trivy generates CycloneDX SBOMs during security scanning,
   and the release job attests the SBOM to the image
1. **Local generation** — Use the `generate-sbom.sh` script for on-demand SBOMs

```bash
# Generate all SBOM formats (SPDX, CycloneDX, table)
./bin/generate-sbom.sh ghcr.io/OWNER/REPO:v1.0.0-minimal

# Generate with vulnerability scan
./bin/generate-sbom.sh --scan ghcr.io/OWNER/REPO:v1.0.0-minimal

# Specific format
./bin/generate-sbom.sh --format spdx-json --output-dir ./sboms ghcr.io/OWNER/REPO:tag
```

## Enforcing in Kubernetes

### Sigstore Policy Controller

The recommended approach for Kubernetes. See
[`examples/security/image-verification/`](../../examples/security/image-verification/)
for policies and setup instructions.

```bash
# Install policy controller
helm install policy-controller sigstore/policy-controller \
  --namespace cosign-system --create-namespace

# Apply image policies
kubectl apply -f examples/security/image-verification/cluster-image-policy.yaml

# Enable enforcement on namespace
kubectl label namespace default policy.sigstore.dev/include=true
```

### OPA Gatekeeper (Registry Allowlist)

For trusted registry enforcement, see
[`examples/security/opa-gatekeeper/`](../../examples/security/opa-gatekeeper/).

## Troubleshooting

### "no matching signatures" error

- **Only release tags are signed.** Branch builds (e.g., `main-minimal`) are
  not signed. Use versioned tags like `v1.0.0-minimal`.
- Check that the `--certificate-identity-regexp` matches the repository.
- Ensure `--certificate-oidc-issuer` is exactly
  `https://token.actions.githubusercontent.com`.

### "no matching attestations" error

- Attestations are added during the release job. Verify the release workflow
  completed without errors in GitHub Actions.
- Use `cosign tree <image>` to inspect what's attached to an image.

### Rekor transparency log lookup

```bash
# Search by image digest
DIGEST=$(crane digest ghcr.io/OWNER/REPO:v1.0.0-minimal)
rekor-cli search --sha "$DIGEST"

# View a specific entry
rekor-cli get --uuid <entry-uuid>
```

### Policy controller rejecting pods

```bash
# Check webhook logs
kubectl logs -n cosign-system -l app=policy-controller-webhook

# Temporarily disable for a namespace
kubectl label namespace my-namespace policy.sigstore.dev/include-
```
